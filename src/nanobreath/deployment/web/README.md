# BreathCoach Browser Demo

Browser demo of the singing breath/phrase coaching system. Supports **live mic recording** with server-side inference.

## Running it

```bash
# Start the inference server (loads NanoPitch + BreathHead in memory at startup)
cd "/Users/festuskahunla/Desktop/Drexel/Spring/Musical AI/Projects/Project 2/prototypes/deployment"
"/Users/festuskahunla/Desktop/Drexel/Spring/Musical AI/Projects/Project 1/NanoPitch/.venv/bin/python" \
    serve.py --port 8421 \
    --nanopitch "/Users/festuskahunla/Desktop/Drexel/Spring/Musical AI/Projects/Project 1/NanoPitch/training/runs/exp12-mixed-aug/checkpoints/best.pth" \
    --breath-head "/Users/festuskahunla/Desktop/Drexel/Spring/Musical AI/Projects/Project 2/prototypes/model/runs/v4-aug-2026-05-18/best.pth"

# open http://localhost:8421/
```

If you just want the static demo without recording support, you can still use
`python3 -m http.server 8421` from inside `web/` — the precomputed clips will work,
but the Record button will fail.

## What it shows

A single-page app with three modes:

1. **Pre-loaded clips** — 5 VocalSet excerpts to demo the model on known data
2. **Record yourself** — click the pink Record button, sing for up to 15 seconds,
   get the same analysis on your own voice. Browser captures mic via Web Audio API,
   encodes WAV in JS, POSTs to `/process`, server returns predictions JSON
3. **Visualization** for whichever clip is loaded:

- **Header**: live model badges — PR-AUC, F1@100ms, inference latency
- **Clip selector**: dropdown of 5 demo clips spanning different singers/styles
- **Log-mel spectrogram** with breath event markers (top: BreathHead, bottom: Ruinskiy baseline)
- **Waveform** with playhead
- **Live coaching**:
  - Current phrase length (ticks up while singing)
  - Last phrase length
  - Total breaths taken
  - Coaching message colored by phrase quality (short/comfortable/long/strained)
- **Breath probability curve** from BreathHead with threshold line
- **Session summary** (after audio ends): phrase count, avg/longest/shortest,
  color-coded bar chart of each phrase, and "vs Ruinskiy baseline" comparison
- **"What is this demo doing?"** expandable explainer with training details

## Regenerating clips

```bash
# Add WAVs to a directory (or replace existing /tmp/demo_clips contents)
python3 ../precompute_predictions.py \
    --nanopitch /path/to/exp12-mixed-aug/checkpoints/best.pth \
    --breath-head /path/to/breath_head/best.pth \
    --audio-dir /tmp/demo_clips/ \
    --output-dir clips/ \
    --threshold 0.25 \
    --phrases-from breath_head
```

The script generates JSON + PNG for each WAV and writes a `manifest.json` the
browser auto-discovers.

## Current model

The header badges reflect the model that was used to precompute the clips:
- **BreathHead**: 15,705 params, 8 hidden channels, kernel 5
- **Trained on**: 75 VocalSet excerpts (32.8 min, 560 weak labels generated by
  Ruinskiy & Lavner 2007), 150 epochs, focal loss (γ=2, α=0.25), seq_len=1000
- **Val PR-AUC**: 0.773, frame F1: 0.67, event F1@100ms: 0.49
- **Inference latency**: ~0.05 ms/frame (PyTorch CPU, single-thread)

## Limitations

- **Server-mediated, not pure-browser inference.** Recording goes through a
  Python server (`serve.py`) which runs PyTorch inference. A pure-browser WASM
  version is the next step — the C inference is already proven bit-identical to
  PyTorch (see `../wasm/test_breath_head.c`). What's missing is emscripten +
  a 2-line patch to NanoPitch's `nanopitch.c` to expose the 384-d concat features.
- **Weak labels only.** The model was trained on Ruinskiy-generated pseudo-labels,
  not hand-labeled gold. Hand-labeling 20+ minutes is the next data-quality step;
  with hand labels we'd expect PR-AUC to push to ≥0.85.
- **Recording max 15s.** Hard-coded cap in JS to keep server processing fast.
  Adjustable in `index.html` (`REC_MAX_SEC`).
- **VocalSet-only training distribution.** Performance on amateur / noisy mic
  input is unmeasured — try the Record button to see how it generalizes.
